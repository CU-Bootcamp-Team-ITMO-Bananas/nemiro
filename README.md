# Nemiro

**Nemiro** — веб-приложение для совместной работы на виртуальной доске в реальном времени. Проект представляет собой быстрый, минималистичный аналог таких инструментов как Miro, FigJam или Excalidraw, с фокусом на скорость работы и простоту использования.

С помощью Nemiro несколько пользователей могут одновременно находиться на одной доске, добавлять стикеры, изображения и текстовые элементы, видеть курсоры друг друга и наблюдать за всеми изменениями в режиме реального времени. Это делает продукт идеальным инструментом для командной работы, мозговых штурмов, планирования и визуализации идей.

Проект построен на современном стеке технологий и использует принципы Clean Architecture на стороне сервера, обеспечивая масштабируемость и поддерживаемость кода.

## Технологический стек

Проект состоит из двух основных частей: клиентского приложения (Frontend) и серверной части (Backend), каждая из которых использует свой набор технологий.

### Frontend

Клиентская часть построена на базе React 18 с использованием TypeScript для типизации. Сборка и разработка осуществляются через Vite, что обеспечивает быструю перекомпиляцию и оптимизированную production-сборку. В качестве основного фреймворка используется React Router DOM для маршрутизации, а стилизация выполнена с помощью Tailwind CSS в сочетании с Radix UI для доступных компонентов.

Для рендеринга холста и работы с графическими элементами применяется библиотека Konva together with react-konva, которая предоставляет высокопроизводительный 2D-рендеринг с поддержкой трансформаций, событий и анимаций. Управление состоянием реализовано через Zustand — легковесную и удобную библиотеку для работы с глобальным состоянием.

Real-time коммуникация с сервером осуществляется посредством Microsoft SignalR, который обеспечивает двустороннюю связь между клиентом и сервером. Аутентификация пользователей выполняется через Telegram Login, что позволяет быстро авторизоваться без необходимости создания дополнительных учетных записей.

### Backend

Серверная часть написана на ASP.NET Core 8 с применением принципов Clean Architecture. Проект разделен на несколько слоев, каждый из которых отвечает за свою зону ответственности.

Слой `NeMiro.Models` содержит модели данных, представляющие сущности системы: доски (Board), элементы (Element), пользователи (User) и указатели курсоров (Pointer). Эти модели используются во всех слоях приложения для передачи данных между компонентами.

Слой `NeMiro.Application` содержит бизнес-логику приложения. Здесь расположены сервисы для работы с пользователями (UserService), досками (BoardsService), элементами (ElementService), указателями (PointerService) и аутентификацией (AuthService). Также в этом слое реализована интеграция с S3-совместимым хранилищем для загрузки изображений через S3Service.

Слой `NeMiro.Infrastructure` отвечает за инфраструктурные аспекты: доступ к базе данных через Repository pattern, миграции и расширения для внедрения зависимостей. Репозитории предоставляют абстрактный интерфейс для работы с данными, скрывая детали реализации доступа к БД.

Слой `NeMiro.Presentation` является точкой входа в приложение. Здесь расположены контроллеры API (AuthController, BoardController, FilesController) и SignalR Hub (BoardHub), который обрабатывает real-time события и обеспечивает синхронизацию состояния между пользователями.

### База данных и инфраструктура

В качестве СУБД используется PostgreSQL — реляционная база данных, которая обеспечивает надежное хранение информации о досках, пользователях и элементах. Для хранения загружаемых изображений применяется S3-совместимое объектное хранилище.

Контейнеризация всего приложения выполняется с помощью Docker, а управление контейнерами осуществляется через Docker Compose. Это позволяет легко разворачивать приложение как в среде разработки, так и в production.

## Архитектура системы

Система Nemiro построена на принципах распределенного взаимодействия, где клиенты общаются с сервером через REST API для долгих операций (аутентификация, загрузка файлов, получение данных о доске) и через SignalR для операций, требующих мгновенной синхронизации (перемещение элементов, обновление курсоров, добавление новых объектов).

### Работа с доской

Когда пользователь открывает доску по еe идентификатору (URL вида `/:boardId`), клиент устанавливает SignalR-соединение с сервером, передавая идентификатор доски и идентификатор пользователя. Сервер добавляет пользователя в группу, связанную с этой доской, и отправляет текущее состояние доски всем участникам группы.

При любом изменении элемента на доске (перемещение, изменение размера, добавление текста) клиент отправляет событие на сервер через SignalR. Сервер обновляет данные в базе данных и рассылает обновленное состояние доски всем остальным участникам группы, обеспечивая синхронизацию в реальном времени.

### Управление курсорами

Каждый пользователь, находящийся на доске, имеет собственный курсор, позиция которого передается на сервер при каждом движении мыши. Сервер хранит текущие позиции всех курсоров и рассылает их обновления другим участникам. Это позволяет видеть, где находятся другие пользователи, и координировать работу над доской.

### Загрузка файлов

При загрузке изображения клиент сначала отправляет файл на сервер через REST API. Сервер передает файл в S3-хранилище и возвращает URL загруженного изображения. После получения URL клиент создает новый элемент типа "изображение" на доске с указанным адресом.

## Возможности

Nemiro предоставляет широкий набор инструментов для совместной работы на виртуальной доске. Система постоянно развивается, и список возможностей регулярно расширяется.

### Работа с элементами

Пользователи могут создавать стикеры — прямоугольные заметки с текстом, которые можно свободно перемещать по холсту. Каждый стикер имеет собственный цвет, который можно изменять через панель конфигурации. Стикеры поддерживают базовое форматирование текста и могут быть удалены клавишей Backspace после выделения.

Изображения загружаются из локального устройства и сохраняются в облачном хранилище. После загрузки изображение автоматически масштабируется до размера, удобного для просмотра, и размещается в центре видимой области холста. Изображения можно перемещать, масштабировать, поворачивать и удалять точно так же, как и стикеры.

Текстовые элементы позволяют добавлять произвольный текст на холст без фона стикера. Это полезно для создания заголовков, аннотаций или пояснений к отдельным частям доски.

### Навигация по холсту

Холст поддерживает неограниченное пространство для размещения элементов. Пользователи могут перемещаться по доске несколькими способами: перетаскивание холста (drag), масштабирование колесом мыши с одновременным удержанием клавиши Ctrl, а также touch-жесты на мобильных устройствах (щипок для масштабирования, два пальца для панорамирования).

Масштабирование позволяет детально рассмотреть отдельные участки доски или увидеть еe целиком. Состояние масштабирования и позиции просмотра синхронизируется между клиентами, позволяя каждому пользователю работать в удобном для него режиме.

### Выделение и трансформация

Любой элемент на доске может быть выделен щелчком мыши. При выделении вокруг элемента появляются маркеры трансформации, позволяющие изменять его размер, поворачивать и перемещать. Панель конфигурации показывает дополнительные настройки выбранного элемента, такие как цвет для стикеров.

Удаление элементов выполняется клавишей Backspace. Система запрашивает подтверждение удаления только если выделен текстовый ввод, в остальных случаях удаление происходит немедленно.

### Создание и доступ к доскам

Каждая доска имеет уникальный идентификатор, который используется в URL. Для создания новой доски достаточно открыть приложение без указания идентификатора — будет создана новая доска с случайным идентификатором. Пользователи могут делиться ссылкой на доску с коллегами, и те смогут присоединиться к совместной работе мгновенно.

## Установка и запуск

### Требования

Для запуска проекта в среде разработки необходимо установить следующие инструменты:

- **Node.js** версии 18 или выше (рекомендуется использовать nvm для управления версиями)
- **pnpm** или **yarn** для управления зависимостями фронтенда
- **.NET SDK** версии 8 или выше
- **PostgreSQL** версии 14 или выше
- **Docker** и **Docker Compose** (опционально, для контейнерного запуска)

### Разработка

#### Запуск базы данных

Самый простой способ запустить PostgreSQL — использовать Docker Compose из директории server:

```bash
cd server
docker-compose up -d
```

Это запустит PostgreSQL контейнер на порту 5430 с учетными данными по умолчанию (postgres/postgres).

#### Запуск сервера

Перейдите в директорию сервера и запустите .NET приложение:

```bash
cd server/NeMiro
dotnet run --project NeMiro/NeMiro.csproj
```

Сервер будет доступен по адресу `http://localhost:5000`. SignalR хаб доступен по пути `/hubs/board`.

#### Запуск клиента

Установите зависимости и запустите dev-сервер:

```bash
cd client
npm install
npm run dev
```

Клиент будет доступен по адресу `http://localhost:5173` (или следующему свободному порту).

Для корректной работы аутентификации через Telegram необходимо настроить переменные окружения или указать параметры бота в конфигурации.

### Docker

Для запуска всего стека приложений через Docker используйте docker-compose в корне проекта. Убедитесь, что у вас настроены все необходимые переменные окружения для подключения к базе данных и S3-хранилищу.

Сборка Docker образов выполняется автоматически через GitHub Actions при пуше в ветку main, однако вы можете собрать образы локально:

```bash
# Сборка клиента
cd client
docker build -t nemiro-client .

# Сборка сервера
cd server/NeMiro
docker build -t nemiro-server .
```

### Production

Для production-развертывания рекомендуется использовать Docker Compose с настроенными переменными окружениями. Клиент обслуживается через Nginx, который также выполняет роль reverse-proxy для API.

Основные переменные окружения для настройки:

- `ConnectionStrings__DefaultConnection` — строка подключения к PostgreSQL
- `S3Settings__AccessKey` — ключ доступа к S3
- `S3Settings__SecretKey` — секретный ключ S3
- `S3Settings__BucketName` — имя S3-бакета
- `S3Settings__ServiceUrl` — URL S3-сервиса

## SignalR Events API

Клиент и сервер общаются через SignalR хаб `/hubs/board`. Ниже описаны основные события и их структура.

### Подключение

При подключении к хабу необходимо передать параметры запроса:

- `board_id` — идентификатор доски
- `user_id` — идентификатор пользователя

После подключения сервер отправляет событие `BoardUpdate` с текущим состоянием доски.

### Входящие события (клиент → сервер)

#### UpdatePointer

Обновление позиции курсора пользователя. Отправляется при движении мыши по холсту.

```typescript
interface PointerUpdateEvent {
  x: number; // Позиция X на холсте
  y: number; // Позиция Y на холсте
}
```

#### UpdateElement

Создание или обновление элемента на доске. Принимает объект элемента с идентификатором и содержимым.

```typescript
interface ElementEvent {
  id: string;
  content: {
    // Содержимое элемента зависит от его типа
  };
}
```

#### DeleteElement

Удаление элемента с доски. Достаточно передать идентификатор элемента.

### Исходящие события (сервер → клиент)

#### BoardUpdate

Основное событие, которое содержит полное состояние доски. Отправляется при любом изменении состояния (новый пользователь, перемещение курсора, изменение элемента).

```typescript
interface Board {
  id: string;
  ownerId: number;
  createdAt: string;
  updatedAt: string;
  elements: BoardElement[];
  users: User[];
  pointers: Pointer[];
}
```

При получении этого события клиент должен обновить локальное состояние, перерисовав все элементы и курсоры других пользователей.

## CI/CD

Проект использует GitHub Actions для автоматической сборки и развертывания. Конфигурация пайплайна находится в `.github/workflows/build-and-push.yml`.

### Процесс сборки

При пуше изменений в ветку `main` автоматически запускается процесс сборки, который выполняет следующие шаги:

1. Сборка .NET приложения и проверка качества кода
2. Сборка Docker образов для клиента и сервера
3. Пуш образов в Docker registry
4. Уведомление Watchtower о доступности новых образов (production и development)

### Настройка secrets

Для работы CI/CD требуется настроить следующие secrets в GitHub:

- `DOCKER_USERNAME` — имя пользователя Docker registry
- `DOCKER_PASSWORD` — пароль или токен доступа к Docker registry
- `WATCHTOWER_HTTP_API_TOKEN` — токен для безопасного API Watchtower
- `WATCHTOWER_UPDATE_ENDPOINT` — endpoint для обновления production версии
- `WATCHTOWER_UPDATE_ENDPOINT_DEV` — endpoint для обновления development версии

### Docker Image Naming

Имена Docker образов автоматически устанавливаются равными имени репозитория. При изменении имени репозитория создается новый образ в Docker registry.
